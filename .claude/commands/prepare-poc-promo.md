## Purpose

Analyzes the POC against the main architecture and data model, identifies production gaps, and generates a template document with placeholders for human decisions. The human fills in the placeholders at their own pace, then runs `/promote-poc`.

**Position in flow:** After `/sync-prd`, before `/promote-poc`.

**What this command does:**
- Invokes `poc-gap-analyzer-agent` to scan POC vs. main architecture for categorized gaps
- Generates `poc/temp/poc_promotion/POC_PROMO_PREP.md` as a **template with `<!-- REQUIRED: ... -->` placeholders**
- The document IS the questionnaire — no interactive Q&A during the command

**What this command does NOT do:**
- Collect decisions interactively (the human fills in placeholders offline)
- Modify any files in `architecture/`, `src/`, or `poc/src/`
- Collect real secrets, API keys, or credentials
- Call external services

## Prerequisites

1. `PRD.md` must exist (ideally post-`/sync-prd`)
2. `architecture/architecture.md` must exist
3. `architecture/data-model.md` must exist (run `/generate-architecture` if missing)
4. `poc/architecture/architecture.md` must exist
5. `poc/src/` must contain POC source code

If any prerequisite fails, ERROR with a clear message and stop.

## Process

### Phase 1: Context Gathering

Read the following files:
1. `PRD.md`
2. `architecture/architecture.md`
3. `architecture/data-model.md`
4. `architecture/modules/*.md` (if they exist)
5. `TECHSTACK.md` (if exists)
6. `poc/architecture/architecture.md`
7. `poc/src/` — scan for mock patterns, dependencies, integrations
8. `poc/package.json` (or equivalent manifest)

### Phase 2: Gap Analysis

**INVOKE `poc-gap-analyzer-agent`** with the following context:

```
PREPARE-POC-PROMO — GAP ANALYSIS:

Analyze the POC against the main architecture and data model.
Identify gaps in the following categories (and ONLY these categories):

1. Data Persistence (DP) — in-memory mocks to replace, entity-to-table mapping,
   missing indexes, enums not yet in DB-native form
2. Authentication/Authorization (AUTH) — mocked vs. real identity, role model, session strategy
3. Secrets Management (SEC) — how API keys / tokens / DB credentials are stored and injected
   (strategy only — never collect real values)
4. External Integrations (INT) — each mocked integration that becomes real
   (which real provider, sandbox availability)
5. Deployment Target (DEP) — hosting platform, runtime, region, environment count

OUT OF SCOPE: observability stack, error-handling/retry defaults, performance targets.

For each gap, provide:
- Gap ID (e.g., DP-1, AUTH-1, SEC-1, INT-1, DEP-1)
- Evidence (specific POC files, mock patterns found)
- Suggested placeholder question for the human
- Suggested options with a recommended default where applicable
- Severity (BLOCKING / ADVISORY)

Input files: PRD.md, architecture/architecture.md, architecture/data-model.md,
architecture/modules/ (if exists), TECHSTACK.md (if exists),
poc/architecture/, poc/src/, and the POC's dependency manifest and config files
(e.g., poc/package.json, poc/pyproject.toml, poc/Cargo.toml, poc/go.mod — whichever
the project uses).
```

**Wait for the agent to complete.** Capture its structured gap list.

### Phase 3: Generate Template

Create `poc/temp/poc_promotion/POC_PROMO_PREP.md` (create the directory if it doesn't exist).

**Document structure:**

```markdown
## Validation Rules

`/promote-poc` will scan this document for remaining `<!-- REQUIRED` markers.
If any are found, it will refuse to proceed and list the unfilled items.
All `<!-- REQUIRED: ... -->` placeholders must be replaced with your decision text.
```

**Rules for generating the template:**
- Only include gaps the agent identified — don't invent categories with no gaps
- If `TECHSTACK.md` or the architecture already answers a question (e.g., DB is already specified), skip that gap — don't ask the human what the architecture already decided
- The "What You Need To Fill In" table at top is a **read-only index** — humans don't modify it
- Humans replace `<!-- REQUIRED: ... -->` markers in the body sections with their decision text
- Each gap gets exactly one `<!-- REQUIRED: ... -->` placeholder
- Never include placeholders for actual secret values — only strategy decisions

### Phase 4: Idempotency

If `poc/temp/poc_promotion/POC_PROMO_PREP.md` already exists:
1. Read the existing file
2. Check if any new gaps were found that aren't in the existing file
3. If new gaps exist: append them to the relevant category section with new `<!-- REQUIRED -->` placeholders
4. Preserve all existing filled-in decisions — never overwrite a human's answer
5. Report what changed: "Added N new gaps. M existing decisions preserved."

## Outputs

- `poc/temp/poc_promotion/POC_PROMO_PREP.md` (created or updated)

## Rules

- **Never write secrets** — only placeholder names and strategy decisions
- **Never call external services**
- **Never modify** `architecture/`, `src/`, or `poc/src/` — read-only reference
- **Only writes to** `poc/temp/poc_promotion/`
- **Skip questions the architecture already answers** — don't ask humans to re-decide
- **Idempotent** — re-running preserves previous human decisions, only adds new gaps

## Next Step

After filling in all `<!-- REQUIRED: ... -->` placeholders, run `/promote-poc` to promote architecture, implement production code, and generate the promotion report.
